生产级幂等性理论框架#

幂等性的数学表达是 $f(f(x)) = f(x)$。在分布式系统中，它的本质是：如何识别重复请求并安全地返回相同结果。

幂等请求的生命周期#

我将解决方案分为三个等级，根据业务成本和重要性进行选择：

第一级：天然幂等（架构设计优化）#

  • 查询（GET）与删除（DELETE）：通常天然幂等（注：删除需根据 ID 删除，而非 DELETE TOP 1）。
  • 绝对赋值型更新：UPDATE score SET value = 100 是幂等的；UPDATE score SET value = value + 10 则不是。
  • 设计建议：尽可能将业务逻辑设计为“指向性”而非“增量性”。

第二级：业务状态机（最推荐，生产首选）#

对于支付、订单等有明确流转状态的业务，利用数据库的原子性：

  • SQL 技巧：使用条件更新。
UPDATE orders SET status = 'PAID', update_time = NOW() 
WHERE order_id = '123' AND status = 'UNPAID';
  • 原理：利用数据库的行锁。如果状态已经变了，Affect Rows 为 0，程序直接返回成功。

第三级：通用幂等表/唯一 Key 机制#

对于没有状态机的操作（如“发帖”、“新增用户”），需要引入 Idempotency Key（幂等键）。

3. 高可用实践方案：通用幂等组件#

这是一套成熟的后端通用方案，通常在 Golang 体系下通过中间件实现。

核心步骤：#

  1. 生成 Key：前端或调用方在请求头带上 X-Idempotency-Key（可以是 UUID 或业务唯一 ID）。
  2. 前置校验（Redis）：
  • 使用 SET key value NX PX 86400000 (Atomic SET if Not Exists)。
  • 如果设置失败，说明请求正在处理或已完成。
  1. 存储结果：
  • 如果处理完成，将响应结果也缓存在 Redis 中。
  • 下次相同 Key 进来，直接从缓存返回结果，不走业务逻辑。

Golang 伪代码实现：#

func IdempotencyMiddleware(next echo.HandlerFunc) echo.HandlerFunc {
    return func(c echo.Context) error {
        key := c.Request().Header.Get("X-Idempotency-Key")
        if key == "" {
            return next(c) // 或者强制要求传入
        }

        // 1. 尝试锁定（原子操作）
        // Status: 0-处理中, 1-已完成
        res, err := redis.Do("SET", key, "processing", "NX", "EX", 3600)
        if err != nil {
            return c.JSON(http.StatusConflict, "Request is being processed")
        }

        // 2. 执行业务逻辑
        err = next(c)

        // 3. 处理完成后更新 Redis 中的状态及返回结果（可选）
        if err == nil {
            redis.Do("SET", key, "completed", "EX", 86400) 
        } else {
            redis.Do("DEL", key) // 失败了允许重试
        }
        return err
    }
}

4. 极端场景下的三道防线#

为了确保“高可用”和“绝对正确”，我会这样布防：

防线 方案 解决的问题
第一道：分布式锁 (Redis/Etcd) SETNX 锁定请求 解决秒级内的并发双击、高频重试。
第二道：数据库唯一索引 (MySQL) UNIQUE KEY (order_sn, source) 底线兜底。即便 Redis 宕机，数据库也会抛出异常，防止产生脏数据。
第三道：乐观锁/版本号 SET version = version + 1 WHERE id = x AND version = old_v 解决分布式环境下旧数据覆盖新数据的问题。

5. 落地建议与避坑指南#

  1. 重试机制的配合：幂等性是为了支持重试。如果后端实现了幂等，调用方（如 MQ）必须配置退避重试策略（Exponential Backoff）。
  2. 存储结果还是返回错误？：对于已处理过的请求，建议返回 上次成功的结果，而非报错。这样调用方会认为任务已完成，停止重试。
  3. 过期时间设置：幂等 Key 的有效期应大于业务处理可能的最大重试周期（通常 24-72 小时）。
  4. 注意原子性：在 Golang 中，如果业务逻辑包含多个 DB 操作，务必开启事务。
