幂等性设计

生产级幂等性理论框架 幂等性的数学表达是 $f(f(x)) = f(x)$。在分布式系统中，它的本质是：如何识别重复请求并安全地返回相同结果。 幂等请求的生命周期 我将解决方案分为三个等级，根据业务成本和重要性进行选择： 第一级：天然幂等（架构设计优化） 查询（GET）与删除（DELETE）：通常天然幂等（注：删除需根据 ID 删除，而非 DELETE TOP 1）。 绝对赋值型更新：UPDATE score SET value = 100 是幂等的；UPDATE score SET value = value + 10 则不是。 设计建议：尽可能将业务逻辑设计为“指向性”而非“增量性”。 第二级：业务状态机（最推荐，生产首选） 对于支付、订单等有明确流转状态的业务，利用数据库的原子性： SQL 技巧：使用条件更新。 UPDATE orders SET status = 'PAID', update_time = NOW() WHERE order_id = '123' AND status = 'UNPAID'; 原理：利用数据库的行锁。如果状态已经变了，Affect Rows 为 0，程序直接返回成功。 第三级：通用幂等表/唯一 Key 机制 对于没有状态机的操作（如“发帖”、“新增用户”），需要引入 Idempotency Key（幂等键）。 3. 高可用实践方案：通用幂等组件 这是一套成熟的后端通用方案，通常在 Golang 体系下通过中间件实现。 核心步骤： 生成 Key：前端或调用方在请求头带上 X-Idempotency-Key（可以是 UUID 或业务唯一 ID）。 前置校验（Redis）： 使用 SET key value NX PX 86400000 (Atomic SET if Not Exists)。 如果设置失败，说明请求正在处理或已完成。 存储结果： 如果处理完成，将响应结果也缓存在 Redis 中。 下次相同 Key 进来，直接从缓存返回结果，不走业务逻辑。 Golang 伪代码实现： func IdempotencyMiddleware(next echo.HandlerFunc) echo.HandlerFunc { return func(c echo.Context) error { key := c.Request().Header.Get("X-Idempotency-Key") if key == "" { return next(c) // 或者强制要求传入 } // 1. 尝试锁定（原子操作） // Status: 0-处理中, 1-已完成 res, err := redis.Do("SET", key, "processing", "NX", "EX", 3600) if err != nil { return c.JSON(http.StatusConflict, "Request is being processed") } // 2. 执行业务逻辑 err = next(c) // 3. 处理完成后更新 Redis 中的状态及返回结果（可选） if err == nil { redis.Do("SET", key, "completed", "EX", 86400) } else { redis.Do("DEL", key) // 失败了允许重试 } return err } } 4. 极端场景下的三道防线 为了确保“高可用”和“绝对正确”，我会这样布防： ...

内存分配

go的内存分配 根据对象大小的不同，go的内存分配有3种类型： Tiny(size < 16B): Objects of size less than 16 bytes are allocated using the mcache’s tiny allocator. This is efficient and multiple tiny allocations are done on a single 16-byte block. Small(size 16B ~ 32KB): Objects of size between 16 bytes and 32 Kilobytes are allocated on the corresponding size class(mspan) on mcache of the P where the G is running. In both tiny and small allocation if the mspan’s list is empty the allocator will obtain a run of pages from the mheap to use for the mspan. If the mheap is empty or has no page runs large enough then it allocates a new group of pages (at least 1MB) from the OS. Large(size > 32KB): Objects of size greater than 32 kilobytes are allocated directly on the corresponding size class of mheap. If the mheap is empty or has no page runs large enough then it allocates a new group of pages (at least 1MB) from the OS. 总结一下Go是如何进行内存分配: mcache中如果有空闲的空间， 那么直接在mcache分配 mcache如果没有，那么尝试去mcentral中获取一个空闲的mspan mcentral中也咩有可用的mspan，那么直接去mheap向操作系统申请可用的mspan ...
