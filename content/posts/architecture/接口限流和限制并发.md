## 接口频率限制 (Rate Limiting) 总结

一套完整的限流方案，本质上是在解决三个问题：**怎么算（算法与实现）**、**卡多细（维度粒度）**、**在哪挡（架构分层）**。

### 一、 核心分类与算法选型

限流从部署维度分为**单机限流**和**分布式限流**，其底层依赖四大基础算法与一种动态自适应机制。

1. **固定窗口 (Fixed Window)**
* **特点**：按固定时间段（如1分钟）计数，实现极简，性能极高。
* **缺点**：存在“临界突刺”问题（例如0:59和1:01分别打满，导致两秒内承受双倍流量）。
* **场景**：对突发流量不敏感、只追求极致性能的粗粒度 RPM/RPH 控制。


2. **滑动窗口 (Sliding Window)**
* **特点**：将固定窗口细分为多个小周期，平滑了临界问题。精度高。
* **场景**：防刷、细粒度 QPS/RPM 控制。


3. **令牌桶 (Token Bucket)**
* **特点**：以恒定速率生成令牌，请求消耗令牌。**允许一定程度的突发流量**。
* **场景**：应对常规互联网业务的峰谷波动，保障用户体验。


4. **漏桶 (Leaky Bucket)**
* **特点**：不管流入多快，流出（处理）速率恒定。属于强力“整形”算法。
* **场景**：保护极度脆弱的下游系统（如核心数据库慢查询、老旧第三方接口）。


5. **自适应限流 (Adaptive Load Shedding)**
* **特点**：抛弃静态死板的 QPS/并发数阈值，基于机器动态负载（CPU > 80%）、请求响应时间（RT）、并发数（Inflight）自动判断是否拒绝新请求。



### 二、 常见 Golang 与 Redis 落地手段

#### 1. 单机并发控制 (单体系统或保护节点自身)

* **基础版：协程信号量**
  利用 Go 非阻塞 channel：`make(chan struct{}, 10)`。每进入一个执行函数写一条，`defer` 消耗一条。极其轻量。
* **进阶版：带权重的并发控制**
  使用官方库 `golang.org/x/sync/semaphore`。支持超时控制和权重分配，避免死锁和携程泄露。
* **频率限制**
  使用 `golang.org/x/time/rate`，它基于令牌桶算法，是标准库生态下的限流利器。或者用 Uber 的 `go.uber.org/ratelimit` 实现漏桶平滑限制。

#### 2. 分布式限流核心 (微服务或集群环境)

* **极简固定时间（排队）**
  利用 Redis 的 `SET EX`。设置过期时间如 250ms，过期后才允许下一个操作。
* **精确并发控制**
  Redis + Lua 脚本。利用原子 `INCR` 增加并发数，操作完毕执行 `DECR` 释放。**Lua 的原子性是关键**。
* **复杂窗口与令牌控制**
  同样依赖 Redis + Lua。滑动窗口通常借助 `ZSET` 结构（用时间戳作为 score），令牌桶则用 Lua 脚本来原子计算时间差并补发令牌。

### 三、 维度的精细化拆分

优秀的架构不应只卡“全局 QPS”，而是要能做到**千人千面**的防控：

1. **IP/设备特征**：结合边缘节点或 WAF，用于防止恶意请求与黑产。
2. **User ID/Tenant (租户)**：SaaS 系统基操，区分 VIP 用户、不同企业客户的调用额度。
3. **API 路由/参数**：高危接口（如大报表导出）单独限流，防止一条烂 SQL 拖垮全站。

### 四、 完整的系统分层架构设计

真实的大流量场景下，防线必须层层递进：

1. **边缘/网络层 (WAF / Nginx)**
* **手段**：基于 Nginx `limit_req` 模块或 OpenResty + Lua。
* **职责**：过滤纯恶意的扫描、异常 IP 高频攻击，直接在网络边界掐断。


2. **网关层 (API Gateway / Envoy)**
* **手段**：分布式限流的核心战场（Redis + Lua）。为追求极致吞吐，可采用“Local Cache + Redis 异步集群同步”机制。
* **职责**：全局业务限流，基于租户、路由做频控。超限时返回 `HTTP 429` 并在 Header 附带重试时间。


3. **微服务层 (Microservices)**
* **手段**：单机限流器 (`x/time/rate`) 或自适应 BBR 限流中间件。
* **职责**：兜住底线，保护微服务自身实例不被压垮，保障核心链路不雪崩。



### 五、 限流触发后的柔性降级 (Fallback)

限流只是手段，最终目的是为了业务“可用”。

1. **Fail-fast (快速失败)**：非核心业务，直接返回错误或系统繁忙提示。
2. **Degradation (优雅降级)**：比如推荐流被限流，那就去查本地缓存拿静态热门数据返回；评论数被限流，就不展示数字或展示假数字。
3. **MQ 削峰 (异步排队)**：合规但超频的突发写请求（如秒杀下单），丢入 Kafka 或 RocketMQ，后端匀速消费，前端转为“排队中”。

## 布式并发控制实现方案总结：
核心逻辑已不再仅仅是“堵住流量”，而是转向“**确定性治理**”和“**削峰填谷**”。
---
### 一、 分布式信号量方案（基于 Redis + Lua）

这是最主流的精细化控制方案，适用于**对全局并发数有严格要求**的场景（如：对接银行接口、控制昂贵的 GPU 计算资源）。

* **实现原理**：
* 利用 Redis 的 Lua 脚本保证“查询-判断-计数”的原子性。
* **不同于限流计数器**：计数器是随时间自动重置，而信号量必须有**获取（Acquire）**和**释放（Release）**过程。


* **生产级细节（如何防止“死锁”）**：
1. **自动过期（TTL）**：由于分布式环境下节点可能宕机，获取名额时必须给对应的 Key 或记录设置超时时间。
2. **唯一标识（Owner）**：释放时必须校验身份（如 RequestID），防止 A 节点任务超时后，名额被系统回收，A 任务完成后又误删了 B 节点正在持有的名额。
3. **心跳续期（Watchdog）**：对于长耗时任务，客户端需异步续期，确保任务没跑完前名额不被回收。



### 二、 集中式限流中心（Token Server / Sidecar）

当集群节点达到成千上万规模时，Redis 的网络开销和热 Key 问题会成为瓶颈，大厂通常采用独立服务或 Sidecar。

* **实现代表**：阿里 Sentinel（集群模式）、Istio (Envoy)。
* **核心机制**：
* **租约模式（Lease）**：服务节点不是每来一个请求就问一次 Server，而是**批量申请**。比如一次性申请 50 个名额缓存在本地 Channel，用完再去申请。
* **自适应调整**：限流中心根据各节点的健康状况（RT、成功率）动态分配名额权重。表现好的节点分配更多并发名额。


* **优点**：极高性能，支持复杂的拓扑限流（如 A+B 两个接口总并发不能超 N）。

### 三、 异步队列化与消费端限速（MQ 模式）

这是大厂解决“瞬间高并发”最彻底的**工程化手段**，将并发控制转化为**速率控制**。

* **实现方案**：
1. **生产端挂载**：请求进来直接塞入消息队列（Kafka/RocketMQ），立即返回“处理中”状态。
2. **消费端控速**：下游服务开启固定数量的 Consumer 协程（Worker Pool）。


* **精妙之处**：
* 通过控制 **Consumer 的并发数**（节点数 × 单机协程池大小），天然实现了全局并发限制。
* **背压控制（Backpressure）**：当并发过高，MQ 堆积达到阈值时，上游生产端通过限流策略拒绝新请求。



### 四、 自适应并发限流（Adaptive Concurrency Control）

这是高阶方案（如 Netflix 的 ConcurrencyLimit、Bilibili 的 Kratos），旨在摆脱硬编码的阈值。

* **算法思路（类似 TCP BBR）**：
* 系统实时监控当前的**最小响应时间（MinRT）**和**最大吞吐量（MaxQPS）**。
* **公式推演**：`理想并发数 = MaxQPS * MinRT`。
* **自动收缩**：如果当前并发增加导致 RT 显著上升，系统会自动下调并发上限，直到 RT 回归正常。


* **优点**：无需人工调优，能够自动应对硬件升级或链路抖动。

### 五、总结：方案对比与选型

| 方案 | 核心技术 | 适用场景 | 关键细节 |
| --- | --- | --- | --- |
| **分布式信号量** | Redis + Lua | 强限制、资源稀缺（如第三方 API） | 必须处理节点宕机后的释放（TTL/续期） |
| **限流中心** | Sentinel / Envoy | 超大规模微服务集群 | 需支持“批量预取”减少网络 RTT |
| **异步 MQ** | Kafka / 协程池 | 耗时任务、可异步化业务（秒杀、导出） | 需关注 MQ 堆积情况与前端交互反馈 |
| **自适应并发** | 动态 RT 监控 | 复杂分布式环境、防止系统雪崩 | 依赖高精度的指标监控系统 |

### 六、 总结性避坑指南

1. **两级防御**：分布式环境下，一定要有“单机防御 + 全局防御”。全局控制用于业务逻辑，单机控制用于防止单点过载，防止 Redis 挂掉后全线崩溃。
2. **非阻塞原则**：在 API 层控制并发时，尽量使用 `TryAcquire`（获取不到立即报错），而不是长时间挂起连接。长时间阻塞会导致 Web Server 的连接池被耗尽。
3. **可观测性**：必须有 Dashboard 看到当前“正在处理数”和“等待排队数”，否则分布式并发控制就是一个黑盒，出问题时极难排查。
